## 再次修复粘贴无反应 + 明确剪切图标应为 E8B6 而非 E8C6
- 粘贴仍无反应的真正根因：
  - `onContextMenuSelect` 里先调用了 `closeContextMenu()`，该方法会把 `contextMenuTarget.value` 设为 `null`
  - 随后 `await doPaste()` 再读取 `contextMenuTarget.value` 时已经是 `null`，所以插入不到任何输入框
- 修复：在 `closeContextMenu()` 之前先用局部变量 `const target = contextMenuTarget.value` 捕获目标元素，并把 `doCut/doPaste/doUndo/doSelectAll` 改成接收 `target` 参数（默认回退到 `contextMenuTarget.value`），确保菜单关闭后仍能操作原输入框
- 剪切图标：
  - 当前代码里已经是 `\uE8B6`（WinUI/Segoe 的 Cut/剪刀图标）
  - 用户说的 `e8c6` 其实是 `复制` 图标（两片纸），不是剪刀；如果修改时把 `\uE8B6` 写错成 `\uE8C6`，就会显示成复制图标
  - 剪切请用 `\uE8B6`，复制用 `\uE8C8`，粘贴用 `\uE77F`

## 修复搜索框内双右键菜单（原生 + 自定义同时出现）
### 问题
- 在搜索框 `<input>` 内右键时，浏览器原生的输入框右键菜单（剪切/复制/粘贴）和自定义 WinUI 右键菜单同时出现
### 根因
- 全局捕获阶段的 `preventDefault()` 对某些浏览器/输入框的原生菜单抑制不完全，或存在事件冒泡导致二次触发
### 修复（双重防护）
1. **WinSearchBox.vue**：在搜索框 `<input>` 上添加 `@contextmenu.prevent`，从源头阻止浏览器原生右键菜单弹出
2. **useContextMenu.js**：新增 `menuLockUntil` 防护锁——每次 `closeContextMenu()` 后锁定 200ms，期间再次收到的 contextmenu 事件直接 `preventDefault + stopPropagation` 吞掉，防止粘贴/复制等操作引发的连锁触发
- 验证：`npm run build` 零错误
